# 变更：默认渲染视图的语义就地编辑（Typora 风格）

## 为什么
- 现在内容区偏“代码编辑器心智”：虽然强，但对写规范/需求的用户来说，阅读与轻量改字成本高。
- 你提出的目标是“像 Typora 一样”：默认就是排版后的 view，双击即可改一小段内容，且**只有必要时**才暴露 Markdown 语法符号。
- 我们还必须满足“尽量保留原文件字节风格”：BOM换行风格、缩进与空格不应被全局格式化重排。

## 变更内容
- Markdown（含 tasks.md）默认展示为“渲染视图”，不再默认进入全屏编辑器。
- 在渲染视图中支持“语义单元双击编辑”：
  - B：双击选中语义单元（词/链接文本/加粗范围/列表项文本等），进入就地编辑。
  - A：仅在必要时显示 Markdown 语法符号（光标所在行、结构被破坏风险区域、或用户显式进入源码编辑时）。
- tasks.md：默认仍为任务看板；在看板/渲染视图中也支持语义就地编辑，并保证与源 Markdown 一致（同源编辑、同源保存）。
- 提供“兜底源码编辑”入口（非主路径）：遇到复杂结构或冲突时可进入源码块编辑，但不作为默认体验。

## 影响
- 受影响规范：visualizer-core（内容渲染与任务文件交互）
- 受影响代码（预期范围）：内容区渲染链路、搜索与选区逻辑、任务看板与文件保存链路
- 依赖与体积：将引入/替换为“文本为真相 + 装饰层”的编辑核心（详见 design.md）
